# Devcoord SQLite Control Plane Draft

> Status: draft
> Date: 2026-03-07
> Scope: NeoMAGI development coordination only; not product runtime
> Related: [`decisions/0050-devcoord-decouple-from-beads-and-use-sqlite-control-plane-store.md`](/Users/zhiliangzhou/devel/Zhiliang/NeoMAGI/decisions/0050-devcoord-decouple-from-beads-and-use-sqlite-control-plane-store.md)

## 1. Purpose

This draft defines the technical shape of a dedicated SQLite-backed `devcoord` control plane after decoupling it from `beads`.

The goal is to keep three boundaries clear:

- `beads / bd` = backlog / issue graph / work tracking
- `scripts/devcoord` = coordination protocol runtime
- `dev_docs/logs/*` + `project_progress.md` = projection / human-readable evidence

This draft focuses on two things:

- the minimal `.devcoord/control.db` data model
- a slimmer `coord.py` command surface that preserves protocol semantics while reducing CLI sprawl

## 2. Non-Goals

- Do not change the product runtime PostgreSQL baseline.
- Do not move product data or memory data into SQLite.
- Do not weaken `AGENTTEAMS.md` protocol requirements.
- Do not reintroduce direct hand-written control state in `dev_docs`.
- Do not turn `devcoord` back into a prompt-only process.
- Do not design for distributed multi-host coordination in this phase.

## 3. Design Principles

- Single-machine shared control plane.
- Standard library first: use Python `sqlite3`, no ORM, no new service.
- Deterministic writes through `scripts/devcoord`, not ad-hoc shell/database access.
- Append-only audit events remain first-class.
- Aggregated state is derived and updated transactionally, not inferred from prose.
- Projection remains disposable and regenerable.
- Human CLI should be small; machine entrypoint can stay structured and explicit.

## 4. Storage Layout

### 4.1 Directory

Use a dedicated control-plane directory at repo root:

```text
.devcoord/
  control.db
  schema_version.json (optional)
```

Rules:

- All worktrees point to the same repo-root `.devcoord/control.db`.
- `.devcoord/` is internal coordination state, not product runtime state.
- `dev_docs/logs/<phase>/...` and `dev_docs/progress/project_progress.md` remain projections generated by `render`.

### 4.2 Why a single SQLite file

- No new server dependency.
- Safe for single-machine multi-worktree coordination.
- Simple backup, inspection, and reset.
- Keeps `devcoord` independent from both `beads` and product PostgreSQL.

## 5. High-Level Architecture

```text
LLM / skill
  -> scripts/devcoord/coord.py
    -> SQLite store (.devcoord/control.db)
    -> dev_docs/logs/<phase>/*
    -> dev_docs/progress/project_progress.md

beads / bd
  -> backlog / issues / epics / review follow-up only
```

Key rule:

- `scripts/devcoord` remains the only protocol runtime.
- SQLite is only the persistence layer for coordination state.
- `beads` is no longer queried or mutated for control-plane state.

## 6. Minimal Data Model

The schema should stay intentionally small. The store needs only six logical object families:

1. `milestones`
2. `phases`
3. `gates`
4. `roles`
5. `messages`
6. `events`

### 6.1 `milestones`

Purpose:

- top-level coordination unit
- one row per milestone, such as `p2-m1a`

Suggested columns:

```sql
milestone_id TEXT PRIMARY KEY
run_date TEXT NOT NULL
status TEXT NOT NULL
schema_version INTEGER NOT NULL
created_at TEXT NOT NULL
closed_at TEXT
```

Notes:

- `status` is coarse: `active` or `closed`
- this replaces the old milestone bead root object

### 6.2 `phases`

Purpose:

- aggregated state per milestone phase

Suggested columns:

```sql
milestone_id TEXT NOT NULL
phase_id TEXT NOT NULL
phase_state TEXT NOT NULL
last_commit TEXT
opened_at TEXT
closed_at TEXT
PRIMARY KEY (milestone_id, phase_id)
```

Notes:

- `phase_state` is aggregated state such as `pending`, `active`, `closed`
- `last_commit` supports projection and resume logic

### 6.3 `gates`

Purpose:

- one row per gate as the aggregated authorization window

Suggested columns:

```sql
milestone_id TEXT NOT NULL
gate_id TEXT NOT NULL
phase_id TEXT NOT NULL
allowed_role TEXT NOT NULL
target_commit TEXT NOT NULL
gate_state TEXT NOT NULL
result TEXT
report_path TEXT
report_commit TEXT
opened_at TEXT
closed_at TEXT
PRIMARY KEY (milestone_id, gate_id)
```

Notes:

- `gate_state` is the authoritative aggregated state: `pending`, `effective`, `closed`
- `report_path` and `report_commit` remain close-time evidence

### 6.4 `roles`

Purpose:

- aggregated liveness and current execution position per role

Suggested columns:

```sql
milestone_id TEXT NOT NULL
role TEXT NOT NULL
agent_state TEXT NOT NULL
action TEXT
current_task TEXT
current_phase TEXT
current_gate TEXT
last_activity TEXT
stale_risk TEXT
PRIMARY KEY (milestone_id, role)
```

Notes:

- replaces the old agent bead state
- supports watchdog and projection output

### 6.5 `messages`

Purpose:

- commands that require ACK semantics

Suggested columns:

```sql
message_id INTEGER PRIMARY KEY AUTOINCREMENT
message_key TEXT NOT NULL UNIQUE
milestone_id TEXT NOT NULL
gate_id TEXT
phase_id TEXT
command_name TEXT NOT NULL
target_role TEXT NOT NULL
target_commit TEXT
requires_ack INTEGER NOT NULL
effective INTEGER NOT NULL
sent_at TEXT NOT NULL
acked_at TEXT
ack_role TEXT
ack_commit TEXT
source_message_key TEXT
payload_json TEXT NOT NULL
```

Notes:

- this table is for `GATE_OPEN`, `STOP`, `WAIT`, `RESUME`, `PING`
- `effective=0` until ACK is accepted
- `payload_json` allows narrow extensibility without growing the top-level schema too fast

### 6.6 `events`

Purpose:

- append-only audit stream

Suggested columns:

```sql
event_id INTEGER PRIMARY KEY AUTOINCREMENT
milestone_id TEXT NOT NULL
event_seq INTEGER NOT NULL
phase_id TEXT
gate_id TEXT
role TEXT
event_type TEXT NOT NULL
status TEXT
task TEXT NOT NULL
target_commit TEXT
result TEXT
report_path TEXT
report_commit TEXT
branch TEXT
eta_min INTEGER
source_message_key TEXT
payload_json TEXT NOT NULL
created_at TEXT NOT NULL
UNIQUE (milestone_id, event_seq)
```

Notes:

- `event_seq` remains a per-milestone monotonic sequence for audit/render ordering
- all protocol evidence lives here
- this table replaces append-only event beads, not projection JSONL

## 7. Transaction Rules

SQLite is only acceptable if the write rules are explicit:

- Enable WAL mode.
- Set a busy timeout.
- Every command runs in a single transaction.
- `event_seq` allocation and aggregated state updates happen in the same transaction.
- `ACK` must atomically:
  - validate a pending message
  - mark it effective
  - append the `ACK` event
  - append the corresponding effective event if required
- `gate-close` must atomically:
  - validate review evidence
  - validate `audit.reconciled=true`
  - update the gate row
  - append the close event

This keeps the current `append-first` and `fail-closed` semantics intact.

## 8. Projection Model

The generated files stay the same:

- `dev_docs/logs/<phase>/{milestone}_{date}/heartbeat_events.jsonl`
- `dev_docs/logs/<phase>/{milestone}_{date}/gate_state.md`
- `dev_docs/logs/<phase>/{milestone}_{date}/watchdog_status.md`
- `dev_docs/progress/project_progress.md`

Rules:

- Projection is regenerated from SQLite, not incrementally hand-maintained.
- `audit` compares SQLite state against generated projection.
- `render -> audit -> read projection` remains the required order before gate close.

## 9. Simplified `coord.py` Command Surface

## 9.1 Problem with the current surface

Current `coord.py` exposes too many flat top-level commands. The problem is not that protocol events are unnecessary; the problem is that every protocol event became a peer CLI concept.

That creates two kinds of noise:

- humans must remember too many verbs
- the CLI shape does not reflect the conceptual grouping of the protocol

## 9.2 Design target

Keep protocol semantics, but reduce the top-level surface to grouped commands.

Target shape:

- current flat top-level surface: 16 commands
- target top-level surface: 7 commands

### Machine-first entrypoint

Keep one structured machine entrypoint:

```text
uv run python scripts/devcoord/coord.py apply ...
```

`apply` should be the default agent-facing interface.

### Human/debug entrypoint

Expose a smaller grouped CLI:

```text
coord.py init
coord.py gate ...
coord.py command ...
coord.py event ...
coord.py projection ...
coord.py milestone ...
coord.py apply ...
```

This reduces the top-level command count without removing protocol expressiveness.

## 9.3 Proposed grouped commands

### `init`

Responsibilities:

- initialize milestone state
- initialize roles
- initialize schema metadata

### `gate`

Subcommands:

- `gate open`
- `gate review`
- `gate close`

Meaning:

- `gate open` is the human-friendly wrapper for issuing `GATE_OPEN`
- `gate review` records review evidence
- `gate close` performs guarded close

### `command`

Subcommands:

- `command ack`
- `command send`

`command send` covers:

- `STOP`
- `WAIT`
- `RESUME`
- `PING`

Notes:

- `GATE_OPEN` is intentionally placed under `gate open`, not generic `command send`, because it is the most common and most semantically important path

### `event`

Subcommands:

- `event heartbeat`
- `event phase-complete`
- `event recovery-check`
- `event state-sync-ok`
- `event stale-detected`
- `event log-pending`
- `event unconfirmed-instruction`

These remain distinct protocol events, but they no longer need to be flat top-level commands.

### `projection`

Subcommands:

- `projection render`
- `projection audit`

### `milestone`

Subcommands:

- `milestone close`

## 9.5 Compatibility strategy

Do not break existing prompts all at once.

Migration path:

1. keep existing flat commands as compatibility aliases
2. implement grouped commands as the canonical form
3. switch skills, runbooks, and PM action plans to grouped form
4. eventually deprecate the old flat aliases

This keeps existing `devcoord` usage stable while shrinking the real public surface.

## 10. Service Layer Shape

The current `CoordService` semantic methods can remain mostly intact.

The main refactor should be:

- introduce a `CoordStore` abstraction
- move `beads`-specific persistence out of the service logic
- add `SQLiteCoordStore`

Recommended split:

- `CoordService`
  - protocol rules
  - validation
  - ordering
  - projection generation
- `CoordStore`
  - milestone/phase/gate/role/message/event persistence
  - transactional writes
  - query helpers for audit/render
- `SQLiteCoordStore`
  - concrete `.devcoord/control.db` implementation using `sqlite3`

This keeps the protocol semantics stable while changing only the store adapter.

## 11. Migration Strategy

### Stage 0: ADR accepted

- accept ADR 0050
- freeze new `beads`-specific expansion in `devcoord`

### Stage 1: Store abstraction

- introduce `CoordStore` interface
- keep the current behavior unchanged

### Stage 2: SQLite implementation

- add `.devcoord/control.db`
- implement `SQLiteCoordStore`
- update `render/audit` to read from SQLite

### Stage 3: Command regrouping

- add grouped CLI commands
- keep old flat aliases
- switch PM/backend/tester skills to grouped or `apply` form

### Stage 4: Cutover

- stop writing control-plane objects to `beads`
- keep `beads` only for backlog tasks
- archive or supersede `beads_control_plane.md`

## 12. Acceptance

- `bd list --status open` is no longer polluted by `coord` control-plane objects.
- `AGENTTEAMS.md` protocol semantics remain enforceable.
- `render` and `audit` still produce the same evidence classes as before.
- `gate open -> ack -> review -> close` works without `beads`.
- restart/resume handshake still works.
- closeout supports `milestone close` without touching backlog issues.
- top-level `coord.py` command groups are fewer and easier to remember than the current flat surface.

## 13. Open Questions

- Should `.devcoord/control.db` be ignored by git, or kept local-only through existing ignore rules plus generated projection evidence?
- Should `schema_version` live only in SQLite metadata, or also in sidecar `schema_version.json` for easier manual inspection?
- Do we want `gate open` to be a semantic alias only, or the canonical path replacing `open-gate` immediately?

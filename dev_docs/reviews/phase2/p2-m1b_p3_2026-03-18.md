## Acceptance Criteria Audit

### 1. PromptBuilder only consumes `ResolvedSkillView.llm_delta` — PASS
`_layer_skills()` (L131-142) checks `skill_view.llm_delta` only. No store access, no `runtime_hints`, no `escalation_signals` consumed. Comment at L134-135 explicitly documents this boundary.

### 2. Physical order correct: Safety -> Skills -> Workspace context — PASS
`build()` L77-86: layers list is `[identity, tooling, safety, skills, workspace, compacted_context, memory_recall, datetime]`. Safety (index 2) before Skills (index 3) before Workspace (index 4).

### 3. Semantic authority annotation exists and is correct — PASS
Docstring at L72-75:
```
Layer physical order: Safety -> Skills -> Workspace context.
Semantic authority: Safety > AGENTS.md > USER.md > skill delta > SOUL.md > IDENTITY.md
Skill delta must not override AGENTS.md or USER.md directives.
```
This matches the design intent: safety is paramount, workspace persona files (AGENTS.md, USER.md) override skill experience, and skill experience overrides SOUL.md/IDENTITY.md.

### 4. `pre-plan` join point at correct position — PASS
In `_initialize_request_state()` (L101-126), skill resolution happens:
- **After**: scope_key resolution (L94), mode resolution (L95), tools schema resolution (L96), compaction state loading (L97), memory recall (L98), contract refresh (L99)
- **Before**: `_build_system_prompt()` call (L128-136)

This is the correct position: mode/scope/tools are resolved, then skills use that context (tool_names, channel, mode) to build TaskFrame, then the system prompt is built with the resolved skill_view.

### 5. `post-run-learning` via unified terminal helper — PASS
`_finalize_task_terminal()` (L587-613) is the single entry point. Called at:
- `_complete_assistant_response()` L490: `terminal_state="assistant_response"` (success=True default)
- `_run_iteration_loop()` L203: `terminal_state="max_iterations"`, `success=False`

### 6. `post-run-learning` allows only 4 terminal_state values — PASS
`TaskOutcome.terminal_state` (types.py L148-154) is typed as:
```python
Literal["assistant_response", "tool_failure", "guard_denied", "procedure_terminal", "max_iterations"]
```
That is 5 values including `procedure_terminal`. Currently only `assistant_response` and `max_iterations` are wired. `tool_failure` and `guard_denied` are noted for P4 integration (L598-599). `procedure_terminal` is schema-only (see item 7).

### 7. `procedure_terminal` is only in schema, not implemented — PASS
`procedure_terminal` appears only in the `Literal` type annotation in `types.py:152`. No code path in `message_flow.py` or anywhere in `src/` invokes `_finalize_task_terminal` with `"procedure_terminal"`. Grep confirms zero runtime usage.

### 8. `pre-procedure` does not exist or is a no-op — PASS
Grep for `pre.procedure` / `pre_procedure` across entire `src/` returns zero matches. No procedure runtime is being run.

### 9. `build()` skill_view parameter defaults to None — PASS
`build()` signature at L58: `skill_view: ResolvedSkillView | None = None`. Backward compatible: all existing callers that don't pass `skill_view` get `None`, and `_layer_skills(None)` returns `""`.

### 10. Compaction rebuild passes skill_view — PASS
`_apply_compaction_result()` (L292-303) calls `_build_system_prompt()` with `state.skill_view` at L302. This means after compaction, the rebuilt prompt retains the skill experience layer.

### 11. `_detect_teaching_intent()` is lightweight rules — PASS
L573-584: Pure string matching against 6 hardcoded signal phrases (3 Chinese, 3 English). No LLM call, no import of external services. `O(n)` where n is number of signals (constant 6).

## Observations (Non-Blocking)

1. **No `test_agent_loop.py` exists** — the gate instructions reference this file but it does not exist in the repo. The AgentLoop integration is tested indirectly via `test_agent_budget_smoke.py`, `test_agent_compaction_integration.py`, `test_agent_identity.py`, etc. The new skill runtime paths (pre-plan, post-run-learning) are exercised only when `skill_resolver` is non-None, and since existing tests construct AgentLoop without skill deps, the code paths are guarded by `if loop._skill_resolver is not None` (L107) and `if loop._skill_learner is None` (L600). No breakage possible on existing flows.

2. **`skill_learner` typed as `object | None`** — agent.py L66 uses `object` rather than a Protocol/ABC. This is acceptable for P3 (placeholder), but P4 should tighten the type to a proper protocol.

3. **`teaching_intent` is computed but not yet consumed** — `RequestState.teaching_intent` is populated (L126) and stored in state (L155), but no downstream code reads it in P3. This is a pre-wired field for P4 learning, which is fine.

## Findings

### F-1: tool_failure/guard_denied terminal states not wired in AgentLoop (LOW)

`_finalize_task_terminal` is called from two sites in `message_flow.py`:
- L203: `terminal_state="max_iterations"` (success=False)
- L490: `terminal_state="assistant_response"` (success=True)

The `tool_failure` and `guard_denied` terminal states are fully supported in `SkillLearner._compute_updated_evidence()` and tested, but there is no call site in the current AgentLoop that passes these terminal states. This means in V1, structured tool failures during an iteration loop do not currently trigger negative evidence recording on resolved skills -- only `max_iterations` (when the loop exhausts) does.

**Impact**: Low. The learner code is correct and ready. The missing wiring is a conscious V1 scope decision -- the current tool loop handles failures by continuing to the next LLM iteration, and the run still terminates via `assistant_response` or `max_iterations`. Adding `tool_failure`/`guard_denied` wiring would require a design decision about whether individual tool failures should immediately terminate the task (currently they don't). This is documented in the plan as a natural P2-M1c or P2-M2 extension point.

### F-2: No test_agent_loop.py file (INFO)

The plan's Acceptance Criterion 7 references `tests/test_agent_loop.py` as a regression file. This file does not exist and apparently never did -- agent loop testing is distributed across multiple test files (`test_agent_tool_parse.py`, `test_agent_tool_context.py`, `test_agent_flush_persist.py`, etc.) and integration tests. No regression issue, just a plan reference to a non-existent test file.

## Residual Risks

1. **Evidence staleness**: Without `tool_failure`/`guard_denied` wiring (F-1), V1 negative evidence accumulation is limited to `max_iterations` scenarios only. Skills that cause consistent tool failures within a loop won't accumulate failure evidence until the loop actually exhausts. Acceptable for V1 conservative strategy.

2. **DB-dependent integration tests not run**: The 65 errored tests require a PostgreSQL instance. Skill-specific DB integration paths (SkillStore with real PG) are not covered in this test run. The mock-based unit tests and FakeSkillStore e2e tests provide good coverage, but real DB atomicity (apply/rollback transactions) is not verified here.

## Findings

### P2-01: Engine `propose()` accepts redundant `kind` alongside `proposal.object_kind`

- **File**: `src/growth/engine.py`, line 49
- **Description**: `GrowthGovernanceEngine.propose(kind, proposal)` takes a
  separate `kind` argument, but `GrowthProposal.object_kind` already carries the
  kind. If a caller passes mismatched values, the engine delegates to the adapter
  looked up via the `kind` parameter, ignoring `proposal.object_kind`. No runtime
  guard validates consistency.
- **Impact**: Low. In P2-M1a scope there is only one onboarded kind (`soul`), so
  a mismatch is impossible in practice. This is a design-debt item for future
  milestones when multiple kinds are onboarded.
- **Recommendation**: Consider adding a consistency assertion or removing the
  redundant `kind` parameter when multiple adapters are introduced.

### P2-02: No dedicated test module for `src/growth/`

- **File**: N/A (no `tests/test_growth*.py` files)
- **Description**: The Gate G0 deliverables did not include a test module. The
  code passes all import smoke tests and lint. Existing test suite (781 unit
  tests) is unaffected (64 errors are pre-existing DB-dependent integration
  tests).
- **Impact**: Low. Gate G0 is a "foundation type layer" gate; runtime behavior
  testing is expected in subsequent gates when the SoulGovernedObjectAdapter is
  integrated.
- **Recommendation**: Gate G1 should include `tests/test_growth_engine.py` and
  `tests/test_growth_policies.py`.

## Residual Risks

1. **Low**: The `kind` parameter redundancy in `engine.propose()` (P2-01) could
   lead to subtle bugs when multiple adapters are onboarded. Recommend adding a
   guard assertion in Gate G1.

2. **Low**: No test module yet (P2-02). Expected to be addressed in subsequent
   gates. The type-layer-only scope of G0 makes this acceptable.

3. **Low**: `PolicyRegistry.get_kind_policy()` raises `KeyError` (not
   `UnsupportedGrowthObjectError`) for an unrecognised kind. Since
   `GrowthObjectKind` is an exhaustive enum, this is unreachable in normal use,
   but future enum extension could surface an unhelpful traceback. Consider
   wrapping in a dedicated error in a future gate.

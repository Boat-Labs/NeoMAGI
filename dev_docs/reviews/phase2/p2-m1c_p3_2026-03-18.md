## 2. Findings

### F1 (BLOCK): `just lint` fails -- complexity guard regressions in test files

`uv run python -m src.infra.complexity_guard check` reports 4 new function_lines regressions, all in integration test files:

| File | Function | Lines | Limit |
|------|----------|-------|-------|
| `tests/growth/test_gc1_integration.py:173` | `test_teach_propose_eval_apply_reuse` | 67 | 50 |
| `tests/growth/test_gc2_integration.py:224` | `test_promote_skill_to_wrapper_tool` | 66 | 50 |
| `tests/growth/test_gc2_integration.py:300` | `test_eval_failure_vetoes_run` | 90 | 50 |
| `tests/growth/test_gc2_integration.py:401` | `test_apply_then_rollback` | 59 | 50 |

These are new regressions introduced by Phase 3. Per CLAUDE.md, test function limit is 50 lines (red line). The tests are well-structured but need to be split into smaller helper functions to meet the complexity budget.

**Fix**: Extract common setup, assertion, and artifact-verification logic into helpers; or split multi-step integration tests into sub-methods called by a short orchestrator.

### F2 (BLOCK): ruff lint errors in test files

5 ruff errors in test files:

- `tests/growth/test_gc1_integration.py`: 2 unused imports (`GrowthEvalResult`, `GrowthLifecycleStatus`), 1 import sort issue (I001)
- `tests/growth/test_gc2_integration.py`: 1 unused variable (`rollback_gv`), 1 import sort issue (I001)

While `just lint` only runs ruff on `src/` (which passes), these are code quality issues that should be fixed. The unused imports and variable indicate incomplete cleanup after development.

**Fix**: Remove unused imports, fix import ordering, use `_ = await engine.rollback(...)` or `await engine.rollback(...)` directly.

### F3 (NON-BLOCK): `CaseRunner.finalize()` status preservation is incomplete

`finalize()` line 153 only preserves `failed` and `rolled_back` statuses:

```python
if run.status in (GrowthCaseStatus.failed, GrowthCaseStatus.rolled_back):
    final_status = run.status
```

Missing from the guard: `vetoed` and `planned`. If `finalize(run_with_status_running, ..., passed=False)` is called, the result would be a finalized run with `status=running`, which is semantically incorrect. The GC-2 eval failure test works only because the caller manually sets `vetoed` before calling `finalize`.

**Recommendation**: Either add `vetoed` to the preservation guard, or change the logic to: `if passed=False and run.status not in terminal states, set status=failed`.

### F4 (NON-BLOCK): No `record_veto()` method on CaseRunner

`CaseRunner` has `record_proposal`, `record_eval`, `record_apply`, `record_rollback`, and `finalize`, but no `record_veto()`. The GC-2 eval failure test manually sets the `vetoed` status via `model_copy`. This is workable but asymmetric with the other lifecycle methods.

**Recommendation**: Add a `record_veto()` method for consistency, or document that veto is handled externally.

### F5 (NON-BLOCK): `_check_promote_conditions()` is a test-local helper, not in production code

The promote condition check (`usage_count >= 3`, `success_rate >= 0.8`) is implemented only as a helper function in `test_gc2_integration.py`, not in any production module. The catalog entry `gc-2` declares `entry_conditions` as strings but there is no runtime enforcement of these conditions.

This is acceptable for Phase 3 (catalog + runner infrastructure), but the gap should be tracked for Phase 4/5 or closeout when the runner actually enforces entry conditions.

## 4. Residual Risks

1. **Entry condition enforcement**: `GrowthCaseSpec.entry_conditions` are declarative strings with no runtime enforcement. The CaseRunner starts any cataloged case without checking conditions. Low risk for P2-M1c (only 2 curated cases), but should be addressed before the catalog grows.

2. **Finalize status gap**: `finalize(passed=False)` with `running` status results in a finalized run that is still `running`. No current code path triggers this, but it could be a foot-gun for future callers.

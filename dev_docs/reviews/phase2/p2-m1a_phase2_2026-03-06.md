## Findings

### P2-F01: `propose()` allows empty string for `new_content` (P2 — low)

- **File**: `src/growth/adapters/soul.py`, line 43-47
- **Detail**: The validation checks `not isinstance(new_content, str)`, which rejects
  `None` and missing keys correctly but allows empty string `""` through. The error
  message says "non-empty string" but the check does not enforce non-emptiness.
- **Impact**: Low. The downstream `EvolutionEngine.evaluate()` catches empty content
  via its `content_coherence` check (`bool(content and content.strip())`), so this is
  defense-in-depth, not a correctness gap. The proposal would be recorded in DB but
  would fail evaluation.
- **Recommendation**: Consider tightening to `if not new_content or not isinstance(new_content, str):`
  for consistency with the error message. Can be deferred to G2.

## Residual Risks

1. **Empty-string new_content** (P2-F01): Relies on downstream `evaluate()` to reject.
   Acceptable given the propose-then-evaluate lifecycle, but tightening the check would
   improve fail-fast semantics.

2. **`rollback(**kwargs)` type looseness**: The adapter's `rollback` signature is
   `(**kwargs: object)` to match the `GovernedObjectAdapter` protocol. Callers must know
   to pass `to_version=N` for soul objects. This is intentional for cross-kind genericity
   but could cause runtime `TypeError` if wrong kwargs are passed. The error would surface
   clearly from `EvolutionEngine.rollback()` as a `TypeError` (keyword-only `to_version`).
   Acceptable for P2-M1a scope.

3. **No tests yet**: Work Package D tests are expected in a subsequent gate. Until then,
   the adapter code is only validated by static analysis and this review.

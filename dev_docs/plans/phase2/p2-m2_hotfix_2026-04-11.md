## 1. Slice A — 请求级写工具断路器（OI-M2-04）

### 问题

Procedure 到 terminal 后，model 退化为 `memory_append` 循环，连续 9 轮无意义 tool call。日志中还有 `memory_append` → `current_time` → `memory_append` 交替模式。`MAX_TOOL_ITERATIONS=10` 是唯一截断，没有更早的确定性阻断。

### 方案

在 `tool_concurrency.py:_run_single_tool()` 中，procedure action 分支**之后**、普通工具执行**之前**，检查 `tool.is_read_only` 和请求级累计计数。

状态存储：`RequestState` 新增 `write_tool_counts: dict[str, int]` 字段（默认空 dict），请求级生命周期。

```python
# tool_concurrency.py — _run_single_tool() 内，普通工具执行前
tool = registry.get(tool_name)
if tool is not None and not tool.is_read_only:
    count = state.write_tool_counts.get(tool_name, 0)
    if count >= WRITE_TOOL_REQUEST_LIMIT:
        return _ToolOutcome(
            tool_call=tool_call,
            result={"ok": False, "error_code": "WRITE_TOOL_REQUEST_LIMIT",
                    "detail": f"Tool '{tool_name}' already executed "
                              f"{WRITE_TOOL_REQUEST_LIMIT} times in this request. "
                              "Stop and respond to the user in text."},
            failure_signal=f"tool_failure:{tool_name}",
        )
    state.write_tool_counts[tool_name] = count + 1
```

语义：
- `WRITE_TOOL_REQUEST_LIMIT = 3`：**允许 3 次执行，第 4 次拒绝**
- **累计**而非连续：`memory_append` → `current_time` → `memory_append` 算 `memory_append` 累计 2 次
- 使用现有 `BaseTool.is_read_only` 属性判定（`current_time`、`memory_search`、`soul_status`、`read_file`、`grep`、`glob` 等已声明只读）
- Procedure action 不受此限制（由 `_run_procedure_action` 分支处理，在断路器检查之前已分流）
- 请求级生命周期，不跨请求

### 影响范围

- `src/agent/tool_concurrency.py`：`_run_single_tool()` 内增加计数 + 断路（~12 行）
- `src/agent/message_flow.py`：`RequestState` 新增 `write_tool_counts` 字段（1 行）
- 不改 `prompt_builder.py`、`procedure_bridge.py`、`BaseTool`

### 不做 completion signal

~~v2 中的 A2 completion signal~~ 已删除。`RequestState` 不跨请求，无法在"下一个用户消息"中检测到"上一个请求有 active procedure"。T10 的确定性修复完全靠断路器。

### 验证

- `test_write_tool_request_limit_triggered`：同一写工具执行 3 次后，第 4 次返回 `WRITE_TOOL_REQUEST_LIMIT`
- `test_write_tool_interleaved_with_readonly`：`memory_append` → `current_time` → `memory_append` → `current_time` → `memory_append` → `memory_append`（第 4 次 memory_append 被拒绝，current_time 不计数）
- `test_write_tool_different_tools_independent`：`memory_append` ×3 + `soul_propose` ×3 各自独立计数
- `test_procedure_action_not_limited`：procedure action 不受断路器限制

## 2. Slice B — PublishTool fail-closed + DelegationTool 返回 available_keys（OI-M2-05 + OI-M2-02 部分）

### 问题

- PublishTool 在 `merge_keys` 为空或全部未命中时仍 `ok=True`，state transition 发生但数据丢失
- Model 不知道 worker result 有哪些 key，只能盲猜

### 方案

两个小改动：

#### B1. DelegationTool 成功返回中增加 `available_keys`

在 `delegation.py` 的成功返回路径中，从 `worker_result.result` 提取 key 列表：
```python
return {
    "ok": True,
    "handoff_id": packet.handoff_id,
    "worker_ok": True,
    "iterations_used": worker_result.iterations_used,
    "available_keys": list(worker_result.result.keys()),  # 新增
    "context_patch": {"_pending_handoffs": current_handoffs},
}
```

#### B2. PublishTool fail-closed（空 keys 和 0 命中均拒绝）

`merge_keys` 为空或全部未命中时，返回 `ok=False`，**不清除 staging**，返回 `available_keys` 让 model 重试：

```python
# 提取 source（merge_worker_result 的数据源）
source = worker_data.get("result", worker_data)

if not merge_keys:
    return {
        "ok": False,
        "error_code": "PUBLISH_EMPTY_MERGE_KEYS",
        "detail": "merge_keys must not be empty",
        "available_keys": list(source.keys()),
        "handoff_id": handoff_id,
    }

visible_patch = merge_worker_result(worker_data, merge_keys, proc_ctx)

if not visible_patch:
    return {
        "ok": False,
        "error_code": "PUBLISH_NO_KEYS_MATCHED",
        "detail": f"None of {list(merge_keys)} found in worker result",
        "available_keys": list(source.keys()),
        "handoff_id": handoff_id,
    }
```

关键语义：**`ok=False` 时不修改 `_pending_handoffs`，staging 保持不变**，state 不转换（`apply_action` step 10 在 `ok=False` 时 stay in current state）。Model 可以用返回的 `available_keys` 重新调用 publish。

### 影响范围

- `src/procedures/delegation.py`：成功路径增加 `available_keys`（1 行）
- `src/procedures/publish.py`：`execute()` 中 merge 前增加 fail-closed guard（~15 行）
- 不改 `merge_worker_result()` 函数本身

### 验证

- `test_publish_empty_merge_keys_fail_closed`：`merge_keys=[]` → `ok=False` + `available_keys`
- `test_publish_no_keys_matched_preserves_staging`：`merge_keys=["nonexistent"]` → `ok=False`，`_pending_handoffs` 未被修改
- `test_delegation_returns_available_keys`：delegation 成功返回包含 `available_keys` 字段

## 3. 执行顺序

```
Slice B (publish fail-closed)  →  Slice A (写工具断路器)  →  回归  →  手工复验
```

Slice B 改动更局部、更确定，先做。Slice A 涉及 message_flow 循环逻辑，稍复杂。

### 回归

- `uv run pytest tests/procedures/ -q`（publish / delegation 新增测试）
- `uv run pytest tests/ -q`（全量）
- 手工复验：重走 B 层 T07~T10，确认：
  - T09 publish 的 merge 不再静默为空（`merge_keys=[]` 被拒绝，返回 available_keys）
  - T10 后发消息 model 不再循环调用 memory_append（第 4 次被断路器截断）

## 4. 不做的事

- 不扩展 `ProcedureView` 暴露 `_pending_handoffs` 内容或结构
- 不添加全局 `memory_append` rate limiting（断路器是请求级的）
- 不添加 `ActionSpec(tool=None)` noop 支持
- 不修改 Worker system prompt 或强制 JSON key 结构
- 不添加 completion signal（`RequestState` 不跨请求，无法实现原设计意图）
- 不新增 `BaseTool` 属性（直接使用现有 `is_read_only`）

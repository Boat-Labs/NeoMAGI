# 0024-m1.5-tool-modes-and-priority-reorder

- Status: accepted
- Date: 2026-02-19

## 选了什么
- 在 `M1` 与 `M2` 之间新增 `M1.5`：可控执行闭环（Tool Modes）。
- 调整 v3 推荐顺序为：`M1.5 -> M2 -> M3 -> M6 -> M4 -> M5`。

## 为什么
- `M1.1 ~ M1.4` 已完成后，系统从“能运行”进入“可委派复杂任务”阶段。
- 若在缺少工具授权边界时直接扩展 `write/edit/bash`，会显著放大风险与行为不确定性。
- 先完成模式化授权与能力面收敛，再进入连续性和记忆阶段，可降低后续返工成本。

## 放弃了什么
- 方案 A：保持 v2 顺序不变，直接进入 M2（会话内连续性）。
  - 放弃原因：复杂执行场景的可控性缺口仍未补齐。
- 方案 B：先扩高能力工具，再补授权边界。
  - 放弃原因：风险前置不足，与安全边界不一致。
- 方案 C：将 M1.5 拆到 M2 之后再做。
  - 放弃原因：会把工具可控性问题带入后续阶段，扩大不确定性。

## 影响
- 下一阶段优先进入 M1.5，先建立 `chat_safe/coding` 工具模式边界。
- M2/M3/M4/M5/M6 的进入顺序按本决议更新。
- 后续若再次调整优先级，应新增决议而非覆盖本条。
